1. The GPU Landscape in 2026

The gap between training and inference has never been wider. Training a frontier model requires thousands of H100s for weeks. Fine-tuning one for your domain requires one (or a few) consumer-grade GPUs and the right techniques.

Fine-tuning is no longer a PhD's privilege. With QLoRA, you can adapt a 70B parameter model on a single 24GB GPU. This changes everything.

Key Insight
Fine-tuning is not retraining. You're not teaching a model new knowledge — you're steering its existing knowledge toward your task. This is why LoRA works with 0.1% of the parameters.

5. Why Fine-Tune? Where It Makes Sense

Before you fine-tune, ask: can prompt engineering solve this?

Fine-tuning is worth it when:

  • You need consistent output format across many queries
  • You have domain-specific knowledge the model doesn't know well
  • You want the model to adopt a specific persona, tone, or reasoning style
  • You're building a specialized classifier that needs context-aware embeddings
  • API calls are too expensive for your use case at scale

Fine-tuning is NOT worth it when:

  • The base model already does the job with good prompting
  • You don't have enough training data (<100 examples — use RAG instead)
  • Your task is purely factual retrieval (update your knowledge base, not weights)
The RAG Alternative
If your model doesn't "know" something, it's usually cheaper and fresher to connect it to a retrieval system (RAG) than to fine-tune. Fine-tuning bakes knowledge into weights — it's slow to update and costly to retrain.

6. Choosing a Model: Quantization Overview

Full-precision models are too large for most GPUs. Quantization reduces weight precision (e.g., 16-bit → 4-bit) with minimal quality loss.

AWQ4~37GB~99%+AWQ
GGUF (Q4_K_M)4~40GB~98%llama.cpp

For fine-tuning with LoRA/QLoRA, use BF16 base models (or FP16 for consumer cards). After fine-tuning, quantize for inference.

7. LoRA Fine-Tuning: The Modern Standard

7.1 Why LoRA Works

Full fine-tuning updates billions of parameters. LoRA observes that fine-tuned models stay close to their pretrained initialization — the "direction" of adaptation is low-rank. Instead of updating W, LoRA freezes W and adds two small matrices A and B. Training only A and B is dramatically cheaper.

# LoRA math
# Full fine-tuning:   Y = W_final · X    (W_final updated)
# LoRA:               Y = (W_0 + BA) · X  (only A and B trained, W_0 frozen)
# If W_0 is d×d, A is r×d and B is d×r, with r << d
# Memory cost: 2dr instead of d² — huge savings

7.2 Key LoRA Hyperparameters

ParamTypical ValueEffect
r8, 16, 32, 64Rank. Higher = more capacity, more params. 16 is usually enough.
lora_alpha2×r (e.g. 32 for r=16)Scaling factor. Keep at 2×r for stability.
lora_dropout0.05 – 0.1Regularization. 0.05 is standard.
target_modulesq_proj, v_proj (minimum)Which attention layers to adapt. Add k_proj, o_proj, gate_proj, up_proj, down_proj for more capacity.
bias"none" (default)Usually keep "none". "all" adds bias trainable params.

8. QLoRA: Fine-Tuning a 70B Model on a Single 24GB GPU

QLoRA = Quantized base model + LoRA adapters. The key insight: backpropagation gradients are cheap (they're stored in FP16), only the forward pass uses the quantized base model. You can fit a 70B model in 35GB of VRAM.

from transformers from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training
import torch

# ── 4-bit quantization config ──────────────────────────────────────────────
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",              # Normal Float 4 — optimal for normally distributed weights
    bnb_4bit_compute_dtype=torch.bfloat16,  # Compute in BF16, store in 4-bit
    bnb_4bit_use_double_quant=True,         # Nested quantization: extra memory saving
)

# ── Load base model in 4-bit ────────────────────────────────────────────────
model_name = "meta-llama/Meta-Llama-3-8B-Instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = tokenizer.eos_token

# device_map="auto" distributes layers across available GPUs/VRAM automatically
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=bnb_config,
    device_map="auto",
    trust_remote_code=True,
)

# Required for QLoRA: tell model to prepare for k-bit training hooks
model = prepare_model_for_kbit_training(model)

# ── LoRA config ─────────────────────────────────────────────────────────────
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)

# Wrap model with LoRA — only LoRA params are trainable
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 41M || all params: 8.03B || trainable%: 0.51%
Why NF4, not FP4?
NF4 (Normal Float 4) is designed for weights that follow a normal distribution — which is true for most pretrained model weights. Using FP4 with NF4-quantized models causes a ~6% quality degradation. Always use bnb_4bit_quant_type="nf4".

9. Full Training Loop: Llama 3 + LoRA

from transformers import TrainingArguments, Trainer, DataCollatorForLanguageModeling
from datasets import load_dataset
import torch

# ── Load and prepare dataset ────────────────────────────────────────────────
# Format: one text per line, or use HF datasets with prompt template
dataset = load_dataset("text", data_files={"train": "./my-data.txt"}, split="train")

# Tokenize with chunking (LoRA needs fixed-length inputs)
def tokenize(example):
    return tokenizer(
        example["text"],
        truncation=True,
        max_length=512,
        padding="max_length",
    )

tokenized_dataset = dataset.map(tokenize, batched=False, remove_columns=["text"])

# ── Training arguments ──────────────────────────────────────────────────────
training_args = TrainingArguments(
    output_dir="./llama3-lora-output",
    num_train_epochs=3,
    per_device_train_batch_size=4,        # adjust based on VRAM
    gradient_accumulation_steps=4,         # effective batch = 4×4=16
    gradient_checkpointing=True,           # trade compute for memory
    optim="paged_adamw_32bit",            # Paged optimizer — avoids OOM on long runs
    learning_rate=2e-4,
    weight_decay=0.001,
    fp16=False,                            # use bf16 if supported
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
    warmup_ratio=0.03,
    lr_scheduler_type="cosine",
    report_to="tensorboard",
    save_total_limit=2,                   # keep only last 2 checkpoints
)

# ── Data collator ───────────────────────────────────────────────────────────
collator = DataCollatorForLanguageModeling(
    tokenizer=tokenizer,
    mlm=False,                             # Causal LM (not masked LM)
)

# ── Train ───────────────────────────────────────────────────────────────────
trainer = Trainer(
    model=model,
    args=training_args,
    data_collator=collator,
    train_dataset=tokenized_dataset,
)

# Disable model's default pad token handling for causal LM
model.config.use_cache = False
trainer.train()

# ── Save adapter only (not the full model) ──────────────────────────────────
model.save_pretrained("./my-lora-adapter")
print("✓ LoRA adapter saved to ./my-lora-adapter")

9.1 Merging and Inference

# Load base model (can be different from training quant)
base_model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# Load LoRA adapter on top
from peft import PeftModel
model = PeftModel.from_pretrained(base_model, "./my-lora-adapter")

# Optional: merge weights (no runtime LoRA overhead)
# This produces a standard model you can share/inference normally
merged_model = model.merge_and_unload()
merged_model.save_pretrained("./my-merged-model")

# ── Run inference ───────────────────────────────────────────────────────────
prompt = "Explain quantum entanglement to a 10-year-old:"
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

with torch.no_grad():
    outputs = model.generate(
        **inputs,
        max_new_tokens=256,
        temperature=0.7,
        top_p=0.9,
        do_sample=True,
    )
    print(tokenizer.decode(outputs[0], skip_special_tokens=True))

12. Production Checklist




  
Before You Deploy





    
  
  • ☐ Quantize if needed — don't serve BF16 at scale unless you need it
    • 
  
  • ☐ Benchmark baseline — measure throughput (tokens/sec) before and after
    • 
  
  • ☐ Set appropriate max_tokens — prevent runaway generation
    • 
  
  • ☐ Temperature = 0 for factual tasks, >0 for creative ones
    • 
  
  • ☐ Implement retries + fallbacks — GPU OOM will happen in production
    • 
  
  • ☐ Log inputs and outputs — you need this for the next iteration
    • 
  
  • ☐ Monitor GPU VRAM — use nvidia-smi dmon in production
    • 
  
  • ☐ Set up eval CI — run your eval set on every fine-tune before deploying
    • 
  
  • ☐ Prepare a rollback — keep the previous adapter checkpoint
    •
