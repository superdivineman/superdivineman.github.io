1. Full Duplex 概述

全双工（Full Duplex），在语音AI领域，指系统能够同时接收用户的语音输入并生成语音输出的能力。与传统的半双工（先听后说）不同，全双工系统需要处理三个核心挑战：

  • 打断处理（Interruption Handling） — 用户可以在系统说话时随时打断，系统必须立即停止当前输出并响应新输入
  • 同时听说（Simultaneous I/O） — 麦克风保持开启的同时，语音合成也在进行；两路音频流必须无冲突地混合或切换
  • 话轮转换（Turn-taking） — 准确判断用户何时讲完（end-of-turn），何时又要继续（backchannel / re-engagement），避免过早或过晚响应

全双工是实现自然对话式AI的核心能力。GPT-4o Realtime、Gemini 2.0 Flash Voice 等端到端模型将这一问题推向了新的技术高度。

为什么全双工很难？
传统级联系统（ASR → NLU → LLM → TTS）天然是半双工的：ASR 需要完整的音频段落才能输出，转录完成后才能推进到下一阶段。要实现全双工，必须将整个流水线改为流式（streaming），同时引入复杂的并发控制逻辑。

3. Benchmark 架构

完整的全双工语音Benchmark流程如下，从麦克风输入到扬声器输出经过多个阶段：

Mic Input → VAD → ASR → LLM → TTS → Speaker
~50ms ~150ms ~80ms ~300ms (首Token) ~200ms
↑ 网络延迟（~50ms），流式传输中持续累加

各阶段说明：

  • VAD（Voice Activity Detection） — 检测用户是否在说话，决定麦克风何时激活和静音
  • ASR（Automatic Speech Recognition） — 将语音转文字；流式ASR可实时输出部分转录结果
  • LLM（Large Language Model） — 理解用户意图，生成回复文本；流式输出可大幅降低感知延迟
  • TTS（Text-to-Speech） — 将回复文本转为语音；流式TTS（如 Codec-based）可边生成边播放
Pipeline 并行化
优化后的架构并非完全串行：VAD 检测到一句话结束时，ASR 已经开始处理当前段落，而 LLM 正在基于已转录的部分结果做预测性推理（Speculative Decoding）。真正的瓶颈往往在于 ASR → NLU 的等待和 TTS 合成延迟。

4. 指标测量方法

4.1 E2E 延迟测量

E2E 延迟定义为从用户说完最后一个词到听到系统回应第一个音节的时间。测量方法：

# E2E 延迟测量伪代码
def measure_e2e_latency(audio_file, system):
    # T_user_end: 用户说完的绝对时间戳（通过 VAD 尾音或人工标注）
    T_user_end = detect_end_of_speech(audio_file)

    # T_system_first_audio: 系统输出第一个音节的时间戳
    output_audio = system.process(audio_file)
    T_system_first_audio = detect_first_syllable_timestamp(output_audio)

    e2e_latency = T_system_first_audio - T_user_end
    return e2e_latency

# 测量 P50 / P95 / P99
latencies = [measure_e2e_latency(f, system) for f in test_set]
print(f"P50: {numpy.percentile(latencies, 50):.0f}ms")
print(f"P95: {numpy.percentile(latencies, 95):.0f}ms")

4.2 MOS 评分标准（ITU-T P.800）

MOS 是主观评分，通过人类评测员对语音质量打分（1-5分）：

MOS 分数质量等级描述
5Excellent和现场语音无法区分
4Good有轻微瑕疵，但不影响理解
3Fair有瑕疵，有一定听觉疲劳
2Poor勉强可懂，理解困难
1Bad无法理解

MOS 评测需要至少 20 个独立评测员，覆盖不同性别、年龄、口音的音频样本。自动化 MOS 预测模型（如 VISQOL、POLQA）可作为快速迭代的 proxy metric，但最终仍需人工评测。

4.3 RTF 计算公式

# RTF = Real-Time Factor
# RTF < 1.0 表示系统能实时处理（即处理速度快于音频播放速度）

import soundfile as sf

audio, sr = sf.read("test_audio.wav")
audio_duration_sec = len(audio) / sr

# 实际测量处理时间
import time
t_start = time.perf_counter()
result = asr_model.transcribe(audio)
t_end = time.perf_counter()

processing_time = t_end - t_start
rtf = processing_time / audio_duration_sec

print(f"RTF: {rtf:.3f}")  # e.g., 0.12 → 比实时快 8x

# 噪声注入脚本（使用 speechbrain 模拟不同 SNR）
import torch
import torchaudio
import os

def add_noise(clean_audio, noise_audio, snr_db):
    # SNR (dB) = 10 * log10(P_signal / P_noise)
    clean_power = torch.mean(clean_audio ** 2)
    noise_power = torch.mean(noise_audio ** 2)
    scale = torch.sqrt(clean_power / (noise_power * 10 ** (snr_db / 10)))
    noisy = clean_audio + scale * noise_audio
    return noisy

7. 全双工实现模式

7.1 流式 ASR + 推测性 TTS

最实用的全双工方案：ASR 和 TTS 均流式输出，中间通过缓冲队列协调：

# 流式全双工伪代码架构
class FullDuplexPipeline:
    def __init__(self):
        self.asr_stream = StreamingASR()
        self.llm_stream = StreamingLLM()
        self.tts_stream = StreamingTTS()
        self.vad = VoiceActivityDetector()

    async def process(audio_chunk):
        # 1. VAD 检测用户是否在说话
        is_speaking = self.vad.detect(audio_chunk)

        # 2. ASR 流式输出（可中断）
        asr_partial = self.asr_stream.push(audio_chunk)

        # 3. 若检测到 end-of-turn，触发 LLM
        if self.vad.end_of_turn_detected():
            llm_stream = self.llm_stream.push(asr_partial)

            # 4. LLM 流式输出 → TTS 流式输出，边生成边播放
            async for token in llm_stream:
                tts_audio = self.tts_stream.push(token)
                await self.speaker.play(tts_audio)

        # 5. 用户打断检测 → 取消当前 TTS
        if user_interrupt_signal:
            self.tts_stream.cancel()
            # 重新处理用户新输入

7.2 并行 vs 顺序流水线

模式延迟复杂度适用场景
顺序（Sequential）最高延迟：所有阶段串行低简单聊天机器人
并行（Parallel）最低延迟：ASR、NLU 可并行高实时打断场景
推测性（Speculative）中延迟：基于 partial ASR 提前响应中生产环境推荐

7.3 缓冲策略：Chunk Size vs 延迟权衡

Chunk = 64ms（极低延迟）延迟最优，但 ASR 准确性下降
Chunk = 160ms（推荐）延迟与准确性最佳平衡点
Chunk = 480ms（高准确）延迟较高，适合安静场景

8. 瓶颈分析

对端到端全双工延迟的拆解分析（以级联方案为例，总 P95 ~780ms）：

ASR ████████~150ms (19%)
NLU ████~80ms (10%)
LLM 首Token ████████████~300ms (38%) ← 最大瓶颈
TTS ████████~200ms (26%)
网络 ██~50ms (6%)
总计 ████████████████~780ms P95
最大瓶颈：LLM 推理
LLM 生成首 Token 的时间（TTFT, Time to First Token）占总延迟的 38%，是全双工系统中最关键的性能瓶颈。流式输出（Streaming）能改善感知延迟，但无法改变首 Token 等待时间。这个问题需要通过模型量化、KV Cache 优化和硬件加速来解决。

9. 优化技术

9.1 Chunked 流式传输（8KB 缓冲）

将音频分块处理，每 8KB（约 160ms@16kHz）送入 ASR 流式推理，避免等待完整句子：

# 8KB chunked streaming ASR（以 Faster-Whisper 为例）
from faster_whisper import FasterWhisper

model = FasterWhisper("large-v3", device="cuda")

def stream_transcribe(audio_stream, chunk_size=8192):
    segments, info = model.transcribe(
        audio=audio_stream,
        beam_size=3,           # 降低 beam_size 换取速度
        vad_filter=True,       # 内置 VAD，减少误触发
        chunk_length_s=30,     # 流式 chunk 最大长度
    )
    for segment in segments:
        yield segment.text   # 实时 yield 部分结果

9.3 TTS 早启（Early TTS Kick-off）

不等到 LLM 生成完整回复，只要置信度足够高就开始 TTS 合成：

# 早启 TTS 策略
async def early_kickoff_tts(llm_stream, tts_model):
    buffer = []
    async for token in llm_stream:
        buffer.append(token)
        text = "".join(buffer)

        # 早启条件：buffer ≥ 5 tokens 且 ends with sentence boundary
        if len(buffer) >= 5 and is_sentence_boundary(text):
            await tts_model.synthesize_and_stream(text)
            buffer = []   # 保留未完成部分继续追加

        # 自然停顿点早启（如逗号、句号）
        if has_pause_marker(token):
            await tts_model.add_silence(pause_duration_ms=200)

9.4 ASR 部分输出加速 NLU

ASR 流式输出的部分结果（partial hypotheses）可以提前触发 NLU 意图分类，减少等待：

# Partial ASR → Early Intent Detection
def partial_intent(partial_text):
    # 轻量意图分类器，基于最新几个词的 embedding
    intent = lightweight_intent_classifier(partial_text[-50:])
    return intent  # "greeting", "question", "command", "stop"

# 若 partial intent = "stop"，可提前终止当前响应，节省 TTS 资源
# 若 partial intent = "greeting"，可提前准备寒暄回复

核心结论
端到端原生全双工模型（GPT-4o、Gemini 2.0）在延迟和话轮转换上大幅领先级联方案，但级联方案在 WER 可控性和部署灵活性上仍有价值。生产系统应根据延迟要求、语音质量和成本综合选型。对于需要低延迟(<500ms P95)的场景，优先选择端到端模型。
